Natural Language to SQL Assistant - Project Explanation

1. Project Summary

This project is an AI-powered Natural Language to SQL Assistant for a retail T-shirt store.
It allows a non-technical user to type a question in plain English, and the system converts
that question into a safe MySQL query, runs the query on the database, and returns the answer
in simple language.

Example:
User asks: "How many White Nike shirts are in stock?"
The app:
- understands the question
- reads the database schema
- creates a SQL query
- executes the query on MySQL
- shows the SQL and result
- writes a human-friendly answer


2. Main Goal of the Project

The goal is to make database access easy for users who do not know SQL.
Instead of writing queries manually, a business user can ask natural language questions and
get answers from the store database immediately.


3. Core Technologies Used

- Streamlit
  Used to build the web interface.

- LangChain
  Used to structure prompts and connect the LLM with the database flow.

- Google Gemini
  Used as the LLM that converts natural language to SQL and also converts SQL results back
  into natural language.

- MySQL
  Stores the business data such as T-shirts, stock, price, and discounts.

- SQLAlchemy + PyMySQL
  Used to connect Python code to the MySQL database.

- python-dotenv
  Loads API keys and database configuration from the .env file.


4. Important Files in This Project

5. Request Flow - End to End

Below is the full flow from user input to final response:

Step 1: User opens the Streamlit app
The app starts from main.py and shows a text box in the browser.

Step 2: User enters a question
Example:
"How many White Nike shirts are in stock?"

Step 3: Streamlit receives the question
The question is stored in the variable named question inside main.py.

Step 4: Environment settings are loaded
The app reads API key and database credentials from the .env file.

Step 5: Setup validation happens
The app checks:
- whether the Gemini API key exists
- whether database settings exist
- whether PyMySQL is installed

If something is missing, the UI shows a warning instead of crashing.

Step 6: Database connection is created
The app builds a MySQL connection string and connects to the configured database using SQLDatabase.

Step 7: LLM connection is created
The Gemini model is initialized using GoogleGenerativeAI.

Step 8: Schema is fetched
The app calls db.get_table_info().
This gets table names, columns, and schema details from the database.

Step 9: Prompt is prepared for SQL generation
The schema and user question are inserted into a prompt.
This prompt tells the model:
- create only one SQL query
- keep it read-only
- use only the available schema
- do not return extra explanation

Step 10: Gemini generates SQL
The model returns a SQL query as plain text.

Step 11: SQL is cleaned
The app removes unwanted text such as:
- SQL:
- SQLQuery:
- markdown code blocks
- extra whitespace

Step 12: SQL is validated
The app checks that the SQL is read-only.
Blocked commands include:
- INSERT
- UPDATE
- DELETE
- DROP
- ALTER
- TRUNCATE
- CREATE
- REPLACE
- GRANT
- REVOKE

Step 13: SQL is displayed
The generated SQL is shown inside the Streamlit app so the user can see what is being run.

Step 14: SQL is executed
The cleaned SQL query is sent to MySQL by db.run(sql_query).

Step 15: Raw result is returned
MySQL returns rows or values such as:
- counts
- prices
- stock totals
- matching records

Step 16: Raw result is displayed
The app shows the raw database result inside the interface.

Step 17: Final natural language answer is generated
The app sends:
- original user question
- generated SQL query
- SQL result
to Gemini again.

The model then writes a simple answer in natural language.

Step 18: Final answer is shown to the user
The answer appears in the Streamlit app under the Answer section.

7. Main Functions in main.py

- get_settings()
  Reads values from .env.

- has_real_api_key()
  Checks whether the API key is real or still a placeholder.

- build_db_uri()
  Creates the MySQL connection string.

- driver_is_available()
  Checks whether PyMySQL is installed.

- get_setup_issues()
  Collects setup problems and shows them in the UI.

- init_database()
  Opens the database connection.

- init_llm()
  Starts the Gemini model connection.

- clean_sql_query()
  Cleans the model output and keeps only the SQL query.

- validate_read_only_query()
  Prevents dangerous SQL commands from being executed.

- generate_sql_query()
  Uses Gemini to convert the natural language question into SQL.

- generate_response()
  Uses Gemini again to convert database results into a human-friendly explanation.

- render_setup_help()
  Shows setup instructions inside the app if configuration is incomplete.

- render_sidebar()
  Shows connection details like DB name, host, model, and driver status.

- main()
  Controls the full app flow.

10. Safety in the Project

This project includes a safety layer before running SQL.

Why needed:
LLMs can sometimes generate harmful or incorrect SQL.

Safety rules used:
- only one SQL statement is allowed
- only read-only SQL is allowed
- dangerous commands are blocked
- schema-aware prompting limits the model to known tables and columns

This protects the database from accidental modification.

14. Common Problems

Problem: API key missing
Reason:
GOOGLE_API_KEY in .env is still a placeholder.

Problem: Database connection fails
Reason:
DB credentials or DB name are wrong, or MySQL is not running.

Problem: PyMySQL missing
Reason:
The required driver is not installed in the active Python environment.

Problem: Query returns no result
Reason:
The question may not match the available data in the tables.

Problem: Model generates unusable SQL
Reason:
LLM output may include extra text or incorrect filtering, which is why cleaning and validation are important.


15. Interview Questions and Answers

Q1. What does this project do?
Answer:
This project converts natural language questions into SQL queries, runs those queries on a MySQL database, and returns the result in a human-readable form through a Streamlit interface.

Q2. Why is Streamlit used in this project?
Answer:
Streamlit is used to quickly build a simple web interface where users can type questions and see the SQL query, raw result, and final answer.

Q3. What is the role of Gemini in this project?
Answer:
Gemini is the large language model that first converts the user question into SQL and then converts the SQL result into a plain English response.

Q4. What is the role of LangChain here?
Answer:
LangChain helps structure prompts, connect the LLM with the application flow, and manage the chain between inputs, prompts, and outputs.

Q5. Why do we need the database schema in the prompt?
Answer:
The schema tells the model which tables and columns exist, so it can generate accurate SQL and avoid inventing columns or tables.

Q6. Why is SQL cleaning required?
Answer:
LLMs sometimes return extra text like "SQL:" or markdown code blocks. Cleaning removes that so the SQL can run directly.

Q7. Why is SQL validation required?
Answer:
Validation protects the database by allowing only read-only queries and blocking destructive commands like DELETE or DROP.

Q8. What tables are used in this project?
Answer:
The main tables are t_shirts and discounts. The t_shirts table stores product information, and the discounts table stores discount percentages linked to products.

Q9. What kind of questions can the system answer?
Answer:
It can answer inventory, pricing, brand, color, size, stock, and discount-related questions, as long as the data exists in the database schema.

Q10. What happens after the user enters a question?
Answer:
The app loads configuration, connects to the database, fetches schema information, asks Gemini to generate SQL, validates the SQL, runs it, and then asks Gemini to explain the result in natural language.

Q11. Why is this project useful for non-technical users?
Answer:
It removes the need for users to know SQL syntax. They can ask business questions in plain English and still get answers from structured data.

Q12. What are the limitations of this project?
Answer:
It depends on the accuracy of the LLM, the quality of the prompt, the database schema, and the available data. It also needs strong validation to avoid unsafe SQL generation.

Q13. What would you improve in this project next?
Answer:
I would add authentication, query history, better error messages, stricter SQL parsing, support for charts, and automated tests for SQL generation and result formatting.

Q14. Why use environment variables?
Answer:
Environment variables keep secrets and configuration outside the code, which is cleaner and safer than hardcoding credentials.

Q15. Why keep only one main application file?
Answer:
Keeping one main application file makes the project easier to maintain, debug, and explain, especially for a small demo or academic project.
